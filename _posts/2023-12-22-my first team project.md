## 오늘의 회고

<hr>

- 오전엔 어제 git&github에 빠져 못 했던 웹개발AtoZ 강좌를 4주차까지 들었다. 데이터베이스와 연결하니 확실히 진짜 웹사이트같고 재밌었다. 오후엔 매니저님, 팀원분들과 얘기를 나누었는데, 매니저님이 팀프로젝트를 하는게 어떻겠냐는 말에 다같이 고민하다가 시작!

- 바로 주제 회의와 투표를 해서 음악 추천 페이지로 정해졌다. 사전 캠프를 오래 참여해 정식적으로 프로젝트를 하는 분들은 팀원 소개 페이지를 만드는 게 과제인데, 우리는 해야하는 팀도 아니고 해볼까..? 하다 시작한거라 주제에 구애받지 않을 수 있어 좋았다.

- 어떤 기능을 구현할지 ui는 어떻게 되어있는지 회의해 큰 틀은 잡았는데, 미니 프로젝트인데 사람은 다섯 명이다보니 역할 분담을 어떻게 해야할지가 난관이었다. 다들 경험 없이 처음 하는 프로젝트라 어떤 기준으로 나눠야 하는 지도 모르겠고… 어제에 이어 윤민아 튜터님께 다섯 명이 몰려가서(ㅋㅋ) 관련 질문을 드렸고, 답변을 받았지만! 9시는 다가오고, TIL은 제출해야하고, 내일은 주말이고. 일단 오늘은 여기까지 하는걸로. 의욕은 앞서는데 어떻게 해야할지를 모르니 답답하다. 나도 진짜 모르는데 너무 나대는 거 아닌가 싶기도 하고… 기획 단계에서도 이런데 본격적으로 시작하면 어떨지 두렵다.

### **고민해봐야 할 것**

- 깃 각자 브랜치 만들건지?  
  문제는 내가 깃허브를 잘 모른다… 브랜치 테스트는 해봤는데 push하고 pull request → 충돌 있으면 해결 → 최종 merge 이 절차가 맞나? 여기서 궁금한건 팀원이 pull request 했을때 내가 충돌을 해결해줄 수 있는지. (된다고 한다.) 이게 되면 깃 관련해선 나만 신경쓰면 되니까 좋긴한데 근데 내가 다 처리하면 너무 경험을 빼앗는건가?

- 분담 후 각자 동시에 작업할지? 시간을 나눠서 각각 작업할지?  
  전자는 당연한거긴한데 어쨌든 웹이 헤더가 있으면 밑에 박스가 있고 그런거니까… 후자는 충돌이 안 일어나서 좋을 것 같은데 팀프로젝트 하는 의미가 없을 것 같다. 모르겠따 오늘은 더 찾아볼 힘이 없다
