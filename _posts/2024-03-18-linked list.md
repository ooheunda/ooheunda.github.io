<img src="/assets/img/240318 todobu erd.png" width="500px" alt="240318 todobu ERD">

쉽지 않네...  
Column이랑 Card는 자유로운 순서 변경이 필요한데, 현재 ERD처럼 priority(order) 하나로만 하진 않을 것 같다. 저렇게 두면 각 레코드들을 하나의 배열처럼 관리하는건데 그렇게 되면 삽입 & 삭제가 매우매우매우 비효율적이다.

내 담당은 Board쪽이지만 다른 분들, 튜터님들과 얘기해보며 몇 가지 방법이 나왔다.

1. 상위 테이블(card라면 column, column이라면 board)에서 order 배열로 관리하는 방법  
   결국 이것도 배열로써 관리하는 관점에선 동일하나 레코드의 order값을 하나하나 변경하지 않고 상위 테이블 하나의 레코드만 수정해도 되기 때문에 그나마 낫다.

2. 링크드 리스트처럼 관리하는 방법  
   순서가 필요한 테이블에 컬럼으로 포인터를 넣는다. 코드를 짜본건 아니라서 될 지 모르겠는데, 결국 head가 무엇인지 알 수가 없어서 포인터를 넣는다면 isHead(bool)나 headCard/nextCard(cardId) 이런 식으로 어쨌든 두 컬럼을 추가해야 할 것 같다. 이렇게 하면 전체 레코드를 변경하지 않고 3개의 레코드만 변경해도 된다.

3. 부동소수점을 이용하는 방법(???)  
    아직 완전히 이해된 건 아닌데... order 컬럼을 넣는 건 동일하고, 카드를 생성할 때마다 order는 2배가 되고 일정 숫자를 넘어서면 배수가 점점 줄어든다. 그리고 카드를 중간에 삽입하게 되면, `(삽입하는 공간의 앞의 카드의 order + 삽입하는 공간의 뒤의 카드의 order) / 2`를 order로 준다.  
   그러다보면 언젠가 카드의 order가 소수가 되는데, 소수의 일정자릿수가 넘어가면 정수보다 데이터의 크기가 커지기 때문에 계속해서 작아지는 건 비효율적인 방법이라고 할 수 있다. 그래서 일정 자릿수가 넘어가게 되면 그걸 정수로 다시 세팅해준다는데 이건 아직 이해가 잘 안된다. 그 세팅의 기준이 뭔데?? 누군가 코드를 짜다오... 이 방법은 지금 하고있는 프로젝트에서 레퍼런스하고 있는 트렐로에서도 사용하는 방법이다. (어떤 고트가 직접 보여줌)

## **Linked List 자료구조**

<hr>

### 배경 지식

- 시간 복잡도: 문제를 해결하는데 걸리는 시간과 입력의 함수 관계. 최악의 경우를 기준으로 계산하며, Big-O 표기법으로 표시한다.

- 공간 복잡도: 문제를 해결하는데에 대한 공간과의 상관 관계. 시간 복잡도에 비해선 중요한 지표는 아니지만, 지나치게 많이 사용하지 않도록 주의한다.

### 배열

<img src="/assets/img/240318 array.png" width="500px" alt="240318 array">

- 메모리의 연속적인 위치에 존재하여 연관된 데이터를 하나의 변수에 grouping해서 관리할 수 있고, 반복문과 결합하면 많은 정보도 효율적으로 처리할 수 있다.

- 조회: O(1)의 조회시간을 가진다. index 값을 주면 해당하는 배열의 값을 바로 리턴하기 때문에 무조건 한 번에 조회할 수 있다.

- 정렬: 어떤 알고리즘을 사용하느냐에 따라 시간 복잡도가 다르다.

- 검색: 첫 요소부터 끝 요소까지 순회(반복문)하기 때문에 일반적으로는 O(N)이다. (시간 복잡도는 최악의 경우가 기준임을 유의!) 정렬이 되어있는 배열이면 O(log N)이 가능하다.

- **삽입 & 삭제**: 특정 요소를 삭제하거나 null로 만들어도 그대로 남아있다. (자바스크립트의 배열은 희소 배열이다.)
  ```javascript
  let arr = [1, 3, 4, 2];
  arr[2] = null;
  console.log(arr); //[1, 3, null, 2]
  ```
  - 메모리 효율성의 감소: 요소가 삭제되어도 공간은 여전히 남아있어 불필요하게 메모리를 차지한다.
  - 오류 가능성 증가: 배열을 순회하거나 데이터를 처리할 때 오류가 발생할 수 있다.
  - 성능 저하: 위 케이스를 고려해서 코드를 짠다면 요소가 쓰레기 값인지 계속 체크해야 한다.
  - 이러한 부작용을 막기 위해 `splice()` 메서드가 있다.
    ```javascript
    let arr = [1, 3, 4, 2];
    arr.splice(2, 1); // 2번째 인덱스부터 1개의 요소 삭제
    console.log(arr); //[1, 3, 2]
    ```
    시간 복잡도는 O(N)이다. 삭제 후 재삽입(빈 값 채우기, 땡기기)하는 과정에서 기존 배열 원소들의 위치를 다시 재정렬 하기 때문이다.

### 연결 리스트

- 배열의 삽입&삭제 단점을 보완하는 자료구조로, 마치 화물칸과 같이 노드(요소)들이 앞뒤 연결고리로 연결되어 있어 유동적으로 떼었다 붙였다 할 수 있다.

  - 맨 앞의 노드를 Head, 맨 뒤의 노드를 Tail이라고 한다.
  - 현재 노드가 가리키는 다음 노드를 포인터라고 하며, Tail 노드의 포인터는 Null이다.

- 조회: 원소의 삽입&삭제에 강점이 있지만, 조회에는 다소 약하다. 시간 복잡도는 O(N)으로 특정 노드를 찾으려면 Head 노드부터 연결고리를 타고 타며 찾아야 한다. 배열의 검색과 같다.

- 링크드 리스트 자료구조를 코드로 나타내면 이렇다.

  ```javascript
  class LinkedList {
    // 링크드 리스트 생성자, head에 시작 노드를 연결한다.
    constructor(value) {
      this.head = new Node(value);
    }

    // 추가 (단순 삽입)
    // next는 포인터를 의미하고, 포인터가 null이면 마지막 노드임을 의미한다.
    append(value) {
      let curr = this.head;
      while (curr.next !== null) {
        //마지막 노드가 될때까지 타고 타고...
        curr = curr.next;
      }
      curr.next = new Node(value); //기존의 마지막 노드의 포인터에 새로운 노드 할당
    }

    // 조회
    getNode(index) {
      let node = this.head;
      for (let i = 0; i < index; i++) {
        //index에 도달할 때까지 타고 타고...
        node = node.next;
      }
      return node;
    }

    // 삽입
    addNode(index, value) {
      const newNode = new Node(value);
      if (index === 0) {
        //head 위치에 삽입하는 경우
        newNode.next = this.head;
        this.head = newNode;
        return;
      }

      const node = this.getNode(index - 1); //삽입할 위치의 앞 노드 가져오기
      const nextNode = node.next;
      node.next = newNode;
      newNode.next = nextNode;
    }
  }
  ```
