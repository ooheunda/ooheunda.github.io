## **트랜잭션의 특징 (ACID)**

<hr>

### Transaction

작업의 완전성을 보장해주기 위해 사용되는 개념으로, 특정한 작업을 전부 처리하거나 전부 실패하게 만들어 **데이터의 일관성**을 보장해주는 기능이다. 작업의 단위를 하나의 쿼리에 종속하는 것이 아닌 여러개의 작업(쿼리)을 묶어 하나의 작업 단위로 그룹화하여 처리하는 작업을 뜻한다.

친구에게 천 원을 송금하는 상황을 생각해보자.

1. 내 계좌에서 1,000원 차감
2. 친구의 계좌에서 1,000원 추가

이 일련의 두 과정은 실패하든 성공하든 동시적으로 일어나야 한다. 1번 작업을 수행하고 2번 작업에서 에러가 나는 경우 A 고객의 1,000원은 그냥 증발해버리는 문제가 생긴다. 이런 부분 업데이트(Partial Update)와 같은 상황을 방지하기 위해 트랜잭션이 도입되었다.

결제 시스템에선 물론이고, 작업 내역을 저장하는 로깅작업, 티켓 예매 시스템 등 데이터의 일관성을 유지해야 하는 다양한 상황에서 사용된다.

### 특징1: 원자성(Atomicity)

트랜잭션 내에서 실행되는 명령들을 하나의 묶음으로 처리하여 내부에서 실행된 명령들이 전부 성공하거나 모두 실패해야한다는 특징으로, 여기서 원자성은 나눠질 수 없는 단일 작업(여러 개의 쿼리를 하나로 묶기)이라는 것을 의미한다.

### 특징2: 일관성(Consistency)

트랜잭션 내부에서 처리되는 데이터의 일관성을 유지해야하는 특징이다. 만약 작업이 성공할 경우 아무런 문제가 발생하지 않고, 실패하더라도 작업을 진행하던 도중 실패한 상태로 데이터를 방치하지 않는다. 트랜잭션 내부 로직 실행 중 하나의 쿼리라도 실패하였다면 **`ROLLBACK`**이 실행되어 성공적으로 완료되었던 쿼리들도 트랜잭션 시작 전 상태로 복구된다.

### 특징3: 격리성(Isolation)

트랜잭션이 실행 중인 경우 다른 트랜잭션에 의해 데이터가 변경되는 것을 방지하는 특징이다. 트랜잭션이 완전히 수행되거나 완전히 수행되지 않은 상태를 외부에서 참조할 수는 있지만, 트랜잭션의 중간 과정/결과를 볼 수 없도록 한다.  
MYSQL에서는 사용중인 DB 오브젝트에 [락(Lock)](#락lock)을 걸어 격리성을 구현하게 되는데, 이것을 자원 잠금(Resource Locking)이라고 부른다. 락을 건 상태는 DB에 접속한 또 다른 클라이언트가 해당 DB를 읽거나 사용할 수 없도록 방지하여 데이터 무결성을 보장시켜준다.

- 동시성(Concurrency): 여러 클라이언트가 동시에 하나의 데이터를 사용 및 공유하는 것. 다수의 사용자가 동일한 시스템을 공유하면서 발생하는 동시 접근 문제를 해결해야 한다.

### 특징4: 지속성(Durability)

트랜잭션이 성공적으로 커밋된 후, 해당 트랜잭션에 의해 생성 또는 수정된 데이터가 어떠한 상황에서도 보존되는 특징이다. 트랜잭션이 완료되면 결과는 데이터베이스에 영구적으로 저장되며, 이후 시스템에 어떠한 문제가 생기더라도 데이터는 손상되지 않아 안정성이 유지된다. 트랜잭션이 성공적으로 완료되면 해당 트랜잭션에 의해 생성 또는 변경된 데이터는 **`COMMIT`** 명령을 통해 영구적으로 저장된다.  
트랜잭션 수행 도중 비정상적으로 종료되더라도 시스템은 트랜잭션 로그를 통해 아직 커밋되지 않은 트랜잭션을 복구할 수 있다.

## **락(Lock)**

<hr>

### Lock

동시성을 제어하기 위해 사용하는 기능이다. 해당하는 데이터를 점유하여 다른 트랜잭션의 접근을 막아 동시성과 일관성의 균형을 맞추기 위해 사용한다. 하나의 데이터를 여러 사용자들이 동시에 변경하려고 할 때(동시성) 락이 존재하지 않는다면 한번에 여러번의 수정이 발생하게 되고, 결국엔 최종 수정값을 인지할 수 없게 되는 상황으로 인해 데이터베이스의 일관성이 깨지게 된다.

### 락의 종류

- 공유 락(Shared Locks) \| 읽기 락(READ Locks)

  ```sql
  START TRANSACTION;
  SELECT * FROM table_name LOCK IN SHARE MODE;
  ```

  - 다른 트랜잭션의 데이터 읽기 허용, 수정 금지
  - **`READ` 전용 락**이라고도 불리며, 해당 락을 사용하는 트랜잭션이 완수되면 해제된다.

- 배타 락(Exclusive Locks) \| 쓰기 락(WRITE Locks)

  ```sql
  START TRANSACTION;
  SELECT * FROM table_name FOR UPDATE;
  ```

  - 다른 트랜잭션의 데이터 읽기, 수정 금지
  - **`WRITE` 전용 락**이라고도 불리며, 다른 트랜잭션의 접근을 금지한다.

### 락킹 수준(Locking Level)

- 글로벌 락(Global Locks) \| 데이터베이스 락(Database Locks)

  ```sql
  FLUSH TABLES WITH READ LOCK;
  ```

  - 데이터베이스 서버에 존재하는 모든 테이블에 락을 걸어 현재 트랜잭션을 제외한 나머지 트랜잭션들이 모든 테이블을 사용할 수 없도록 만든다.
  - 가장 높은 수준, 가장 큰 범위의 락이다.

- 테이블 락(Table Locks)

  ```sql
  LOCK TABLES table_name READ;
  ```

  - 다른 사용자가 작업중인 테이블을 동시에 수정하지 못하도록 한다.

- 네임드 락(Named Locks)

  ```sql
  -- 10초 동안 획득하지 못한다면 NULL 반환
  SELECT GET_LOCK('string', 10);
  ```

  - 테이블, 테이블 행과 같은 DB 오브젝트가 아닌 특정한 문자열을 점유한다.

- 메타데이터 락(Metadata Locks)

  ```sql
  -- 테이블 구조 변경 시 MySQL은 내부적으로 메타데이터 락을 설정한다.
  ALTER TABLE table_name ADD COLUMN column_name Int;
  ```

  - 다른 사용자가 작업중인 테이블의 동일 행 및 동일 데이터베이스의 객체를 동시에 수정하지 못하도록 한다.

- **_교착 상태(Dead Lock)_**: 여러 테이블에 락(Lock)을 잘못 적용하여 서로가 서로의 리소스를 점유하고 있어 무한정 기다리기만 하는 상태.

### 격리 수준(Isolation Level)

여러 트랜잭션이 동시에 처리될 때 다른 트랜잭션에서 변경 및 조회하는 데이터를 읽을 수 있도록 허용하거나 거부하는 것을 결정하기 위해 사용하는 것이 격리 수준이다. **데이터의 일관성**과 **동시성 처리 성능** 사이에서 균형을 잡는 것이 중요하다.

- **`READ UNCOMMITTED`**

  - 커밋 되지 않은 읽기를 허용하는 격리 수준
  - 가장 낮은 수준이며 락을 걸지 않아 동시성이 높지만 일관성이 쉽게 깨질 수 있다.

- **`READ COMMITTED`**

  - 커밋 된 읽기만을 허용하고, `SELECT`문 실행 시 공유락을 건다.
  - 다른 트랜잭션이 데이터 수정 중엔 데이터를 읽을 수 없어 커밋되지 않은 읽기현상이 발생하지 않는다.
  - 커밋되지 않은 읽기(Uncommitted Read): 다른 트랜잭션에 의해 작업중인 데이터를 읽게 되는 것. 의도치 않은 데이터를 참조하게 되어 데이터의 일관성이 깨진다.

- **`REPEATABLE READ`**

  - 읽기를 마쳐도 공유락을 풀지 않으며, 트랜잭션이 완전히 종료될 때까지 락을 유지한다.
  - 공유락이 걸린 상태에서 데이터 수정은 불가능하나 데이터 삽입은 가능하기 때문에 팬텀 읽기가 발생할 수 있다.
  - 팬텀 읽기(Phantom Read): 트랜잭션 실행 중 다른 트랜잭션에 의해 삭제된 데이터를 팬텀 행(Phantom Rows)이라고 하는데, 여기서 팬텀행 데이터를 읽는 것을 팬텀 읽기라고 한다.

- **`SERIALIZABLE`**
  - 데이터를 읽는 동안 다른 트랜잭션이 해당 데이터를 읽거나 삽입할 수 없고 새로운 데이터 추가도 불가능하다.
  - 가장 높은 수준이며 동시성이 떨어지는 문제점이 존재한다.
