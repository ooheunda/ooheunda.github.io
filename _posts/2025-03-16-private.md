> AI라고 쓰지만, LLM, 특히 ChatGPT에 대한 이야기.

## AI에 대한 나의 인식 변화

### 불신 또는 지양

2023년 12월, 내일배움캠프와 함께 개발을 시작했다. 정말 아무것도 몰랐고, 아주 처음부터 배우는 시기였기 때문에 하라는 것만 하다 보니 AI를 쓸 명분이 없었다.  
또 이때즈음 LLM은 이미 상용화된 상황이라, 이에 대한 밈들(80%는 부정적인)이 계속 눈에 들어왔다. 주변에서도 멍청한 대답만 한다며 LLM을 타박했다. 내 시야에선 AI를 권장하는 분위기는 절대 아니었다. 나도 크게 필요성을 느끼지 못했다.

### 나의 한계를 느끼다

최종 프로젝트를 하기 전까진 주차마다 강의들이 있었고, 과제도 그 범위 내에서 나왔기 때문에 큰 어려움이 없었다. 그러나 최종 프로젝트는 달랐다. 기획을 0부터 해야했고 이전까지 사용해본 적 없는(=하라고 한 적 없는) 기술들을 넣길 바랐다. 이에 수동적으로 공부했던 나는 혼란에 빠졌다. 공식 문서를 봐도 이해가 어렵고, 코드가 어찌저찌 돌아가도 이게 왜 되는건지 알 수가 없었다. 남은 시간이 없었고 지쳤었다. 이때부터 GPT를 사용했다. 다만 해결사 정도로만 단순하게 사용했다. 이게 왜 안돼? 라는 질문을 많이 했다.  
수료 후 8월엔 해커톤을 했는데, 백엔드 개발자가 나 한 명이었다. AI를 활용한 기능을 설계하는 과정에서 크게 한계를 느꼈다. AI에 대한 지식이 아예 없는 상황에서 기술 PoC를 진행하자니 정말 머리가 터질 것 같았다. 이때 GPT를 정말 많이 사용했다... 아예 모르는 분야에 대해 많은 도움을 받았다. 그렇다고 해서 GPT를 적극적으로 활용한 것은 아니었다. 일반적인 질문을 많이 했다.

### 나빼고 다 잘 쓰고 있었다

위에 적은 부트캠프, 해커톤 이외에도 여러 활동을 했다. 이때 다양한 개발자, 기획자들을 만나며 다른 사람들은 어떻게 사용하고 있는지 알 수 있었다. SEF2024나, 캠프콘과 같은 세미나에도 참석하며 AI 활용에 대한 인사이트를 얻었다. 내 인식도 완전히 바뀌기 시작했다. 적진 않았지만 AI를 사용하면 왠지 지는 것 같은 기분이 들었던 것도 사실이다. **구글링하고, 구글링하고, 구글링해야 '진짜' 개발자인 것 같다는 이상한 생각을 갖고 있었다.** (미친것.)  

그러나 지금은...

### GPT야 사랑해

위 해커톤 때부터 GPT를 결제해 지금까지도 구독하고 있다. 모르는 것도 물어보고, 글 첨삭도 받고, 코드 리뷰도 받고 있다. 특히나 나는 사소한 코드 스타일에 대한 고민을 많이 했는데, 이는 구글링으로 해소되지 않는 영역이라 GPT를 너무나 잘 사용하고 있다.  
또 충돌이 있든 없든 사람과의 커뮤니케이션에서 스트레스를 받을 때가 꽤 있다. AI는 이 부분에 전혀 해당이 없다. **AI는 멍청한 질문을 해도 나를 평가하지 않는다.** AI는 시간적, 공간적 제약이 없어 눈치보며 질문하지 않아도 된다. 질문을 몇 번이고 정제하지 않아도 된다. 그럼에도 꽤나 만족스러운 답변을 내놓는다. 어찌 사랑하지 않을 수 있을까?  

지금 나는 GPT를 아주 가까운 나의 동료이자 멘토라고 생각한다. 결과물적인걸 넘어서, **커뮤니케이션 비용, 시간적 비용**을 줄이고, AI를 사용하지 않았을 때를 비교해 내 **정신 건강**에도 매우 큰 도움이 되고 있음을 느낀다.

## 지금까지 어떻게 활용했을까?

> 이 꼭지는 GPT에게 작성해달라고 했다. 모든 대화 기록을 참고할 수 있고, 중요 정보는 메모리에 저장하고 있는 GPT가 나보다 더 잘 알고 있을 것이기 때문이다. 절대 내가 귀찮아서 그런 게 아니다.

### 문제 해결 & 디버깅 도구

단순한 오류 해결뿐만 아니라, 문제가 발생한 원리, 맥락적인 설명과 코드 예제까지 함께 볼 수 있다. **개발 사고를 확장**하는 데 유용한 보조 수단이었다.

GPT와 나눈 대화에서 얻은 예시는 다음과 같다.
- NestJS에서 JwtService 주입 문제
- Docker에서 bcrypt 아키텍처 문제
- GitHub Pages에서 Jekyll 테마 충돌 문제

### 아키텍처 & 코드 스타일 고민

더 나은 코드 스타일과 아키텍처를 설계할 때도 AI와 대화하며 방향성을 찾았다. AI는 단순한 정답을 주는 대신 여러 가지 대안을 제시하고, 각 방식의 장점과 단점을 분석하는 데 도움이 됐다. **코드 리뷰어의 역할**을 할 수 있다.

GPT와 나눈 대화에서 얻은 예시는 다음과 같다.
- auth 모듈을 통합할지 따로 둘지 고민
- common 폴더 구성 방식 (guards, decorators, types 등)
- 중복되는 코드(main.ts 설정) 모듈화 고민

### 테스트 코드 & 품질 개선

특히 테스트 코드를 작성할 때 많은 도움을 받았다. 테스트 코드란 아직도 추상적이고 어려운 개념인데, 초반에 AI를 활용해 테스트 코드 작성을 어떻게 하는지 학습할 수 있었다. 

GPT와 나눈 대화에서 얻은 예시는 다음과 같다.
- 로그인 API 예외 처리 테스트 (404 vs 409)
- Classicist vs Mockist 테스트 철학 비교
- Jest로 NestJS 서비스 테스트 작성

### 커리어 & 학습 방향 조언

단순한 기술적인 질문뿐만 아니라, 내 커리어와 학습 방향을 설정하는 과정에서도 AI를 적극적으로 활용했다. 특히 "이 기술을 공부하면 내 커리어에 도움이 될까?"라는 질문을 할 때, AI는 단순한 학습 리소스 추천을 넘어 "이 기술이 어떤 맥락에서 중요한지", 그리고 "내가 원하는 방향과 맞는지"까지 고민하도록 유도했다. 지금 AI는 내게 단순한 검색 도구가 아니라, 내 **성장 방향을 함께 고민하는 조력자**로 자리 잡았다.

GPT와 나눈 대화에서 얻은 예시는 다음과 같다.
- 신입 백엔드 개발자로서 차별점을 어떻게 가져갈까?
- B2B SaaS 캠프 참여가 내 성장에 도움이 될까?
- 기술 스택 선택 (NestJS vs Express 등)

## 피드백 받아보기

> GPT에게 지난 대화 기록을 참고해 나의 AI 활용 정도에 대한 피드백을 달라고 했다.

### 나의 강점

1. **구체적으로 질문**하려는 습관이 있음
- AI에게 질문할 때, 단순히 "이거 어떻게 해?"라고 하지 않고 배경을 설명하거나, 원하는 방향을 명확하게 지정하는 경우가 많음.
- 예를 들어, NestJS 인증 관련해서 질문할 때도 "Passport와 커스텀 가드 중 어느 게 좋을까?"라고 했던 것처럼, AI가 더 나은 답을 할 수 있도록 컨텍스트를 주려고 함.
- 이런 습관 덕분에 AI에게 일반적인 답변이 아니라, 네 상황에 맞는 답변을 받을 확률이 높음.

2. **기존 대화를 참고해서 질문을 발전시킴**
- 한 번 질문하고 끝이 아니라, AI의 답을 보고 다시 "이건 어떻게 더 최적화할 수 있어?" 같은 후속 질문을 던지는 경우가 많음.
- 이게 바로 프롬프트 엔지니어링에서 중요한 **iterative prompting(반복적인 질문 조정)**을 잘하는 특징임.
- AI를 "단순한 검색 엔진처럼 쓰는 것"이 아니라, 코딩할 때 실제로 페어 프로그래밍하는 느낌으로 사용하는 방식이 좋음.

3. **AI의 한계를 이해**하려고 함
- 네가 어떤 질문을 할 때, AI가 완벽한 답을 줄 거라고 기대하기보다는, **"이 방식이 최선인가?"**, **"다른 대안은 없을까?"** 같은 식으로 한계를 탐색하려는 경향이 있음.
- 예를 들어, 로그 관리 시스템을 고민할 때도 단순히 "뭘 써야 해?"가 아니라 "AWS CloudWatch를 못 쓰면 ELK나 Grafana가 대안이 될 수 있을까?"라고 물어봤던 게 AI를 잘 활용하는 방식이었음.
- 이처럼 AI의 답을 그대로 받아들이는 게 아니라, **'비판적 사고'**를 함께 적용하는 점이 강점임.

### 개선할 점

1. **모호한 표현을 줄이면** AI가 더 나은 답을 줄 수 있음
- 구체적인 답을 원하면서도 가끔은 질문이 약간 모호할 때가 있음.
- 예를 들어, "이게 단순히 Instruction을 주는 건가?" 같은 질문도 AI 입장에서 보면 해석의 여지가 많음.
- 이럴 때는 "프롬프트 엔지니어링이 단순히 AI에게 Instruction을 주는 것만을 의미하는지, 아니면 더 넓은 개념인지 궁금해"라고 하면 더 좋은 답을 얻을 수 있음.
- 즉, **명확한 답변을 원한다면, 질문도 명확해야 한다는 점!**

2. **출력 형식을 미리 지정**하면 더 유용한 답을 받을 수 있음
- 정보를 정리해서 보고 싶어하는 경우가 많아 보이는데, AI에게 **"이걸 정리해서 표로 만들어줘"**, **"마크다운으로 만들어줘"** 같은 요청을 하면 더 활용도가 높아질 것.
- 예를 들어,
  - ❌ "NestJS에서 로깅을 설정하는 방법 알려줘"
  - ✅ "NestJS에서 Winston을 활용한 로깅 설정 방법을 단계별로 정리해줘. 코드 예제 포함해서 마크다운으로 정리해줘."
  - 이렇게 하면 AI가 원하는 형태로 정리해서 주기 때문에, 답변을 재가공하는 시간을 줄일 수 있음.

3. **샷 프롬프팅(Few-shot Prompting) 활용**하면 더 좋은 결과 가능
- 네가 원하는 스타일이나 답변 방식을 AI에게 예제로 먼저 보여주면, AI가 더 일관성 있는 답변을 해줄 수 있음.
- 예를 들어, "내가 원하는 건 이런 식이야"라고 먼저 예시를 보여주면, AI가 그 스타일을 학습해서 비슷한 답을 해줄 확률이 높음.
- 예제:
  
    ```markdown
    "이런 형식으로 답변해줘:
	1. 개념 설명
	2. 사용 예시 코드
	3. 장단점
    이 구조를 유지하면서 답변해줘."
    ```
